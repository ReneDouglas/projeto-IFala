Sobre o IFala

O IFala é uma plataforma de comunicação segura e anônima voltada para instituições de ensino, com foco em permitir que alunos relatem situações de assédio, violência, discriminação, bullying ou qualquer outra ocorrência que impacte o ambiente escolar/acadêmico.

O sistema oferece um fluxo completo de registro, triagem, acompanhamento e notificação envolvendo alunos e servidores coordenadores, garantindo:

  • Anonimato e proteção dos denunciantes.
  • Registro estruturado das denúncias.
  • Acompanhamento do status pela parte denunciante.
  • Painel de gestão para servidores responsáveis.
  • Auditoria e rastreabilidade das ações realizadas.
Objetivo principal: criar um canal oficial, seguro e rastreável para denúncias internas, reduzindo subnotificações e apoiando decisões institucionais baseadas em dados.

Primeiros passos

Esta seção orienta novos colaboradores sobre como configurar o ambiente de desenvolvimento e entender rapidamente os módulos do sistema.

1. Requisitos de ambiente

  • Git e acesso ao repositório do IFala (GitHub/GitLab).
  • Docker e Docker Compose instalados.
  • JDK 17+ (confirmar a versão exata utilizada no projeto) para o backend.
  • Node.js (LTS) e pnpm/yarn/npm para o frontend (ver seção de tecnologias).
  • IDE/Editor: IntelliJ IDEA / VS Code / outro.

2. Clonando o projeto

git clone <URL_DO_REPOSITORIO_IFALA>
cd projeto-IFala

Consulte o README principal do repositório para detalhes adicionais de configuração.

3. Subindo a stack via Docker

O projeto disponibiliza uma orquestração via docker-compose que sobe backend, banco de dados e serviços auxiliares (como Prometheus, Grafana, etc., dependendo da configuração atual).

# subir todos os serviços em modo detached

Após o build, verifique se os containers essenciais estão saudáveis:

Para o aluno

O aluno acessa o IFala para registrar denúncias e acompanhar o andamento das ocorrências que reportou. O foco é garantir facilidade de uso e sensação de segurança/anonimato.

Como realizar uma denúncia

  1. Acessar a URL pública do IFala informada pela instituição (ex.: portal do IFala).
  2. Selecionar a opção “Registrar denúncia” na tela inicial.
  3. Preencher os campos obrigatórios, como categoria da ocorrência, descrição detalhada, local/data aproximada, pessoas envolvidas (quando aplicável) e anexos (se disponível).
  4. Confirmar se deseja enviar a denúncia de forma anônima ou identificada (conforme política da instituição).
  5. Enviar a denúncia e guardar o código de acompanhamento gerado pelo sistema.

O código de acompanhamento é a chave para que o aluno acompanhe a evolução da denúncia sem necessidade de login.

Como funciona o acompanhamento da denúncia

O acompanhamento pode ser feito pela tela pública de “Acompanhar denúncia”, onde o aluno informa o código gerado ao final do registro:

  • O sistema exibe o status atual (ex.: Recebida, Em análise, Em andamento, Concluída, Arquivada).
  • São mostrados registros de movimentações realizadas pelos coordenadores, preservando a identidade do servidor.
  • Dependendo da configuração, o aluno pode receber notificações por e-mail sobre mudanças de status (via módulo de notificações).

Para o servidor coordenador

Servidores coordenadores são responsáveis por acessar o painel interno, visualizar denúncias recebidas, atuar sobre elas e registrar as ações tomadas.

Realizando o login

  1. Acessar a área restrita do IFala (ex.: /admin ou rota equivalente).
  2. Informar e-mail institucional e senha cadastrada.
  3. Em caso de primeiro acesso, seguir o fluxo definido de “primeiro login” (definir senha, aceitar termos etc., se existir).

Realizando o cadastro de um novo servidor

O cadastro de novos servidores coordenadores geralmente é feito por um usuário com permissão elevada (Administrador/Coordenador principal):

  • Acessar o menu de “Gestão de usuários”.
  • Clicar em “Adicionar servidor”.
  • Preencher nome, e-mail institucional, campus/setor e perfil de acesso.
  • Definir se o usuário receberá e-mail automático de criação de conta para definir senha.

Como funciona o painel de denúncias

O painel interno apresenta:

  • Lista de denúncias com filtros por status, data e categoria.
  • Acesso ao detalhe da denúncia, incluindo histórico de movimentações.
  • Ações disponíveis: atribuir responsáveis, mudar status etc.

Como funciona o acompanhamento de uma denúncia

Para cada denúncia, o servidor pode registrar atividades (comentários internos, encaminhamentos, anexos) e alterar o status do registro, o que eventualmente dispara notificações ao aluno.

Arquitetura e Serviços

Arquitetura Geral

O IFala segue uma arquitetura em camadas, com separação clara entre frontend, backend, banco de dados e serviços de observabilidade.

[ Cliente (Aluno / Servidor) ]
  ↕ HTTP (HTTPS recomendado)
[ Frontend (React + Vite + TypeScript) ]
  ↕ REST API / JSON
[ Backend (Spring Boot) ]
  ↕ JDBC / ORM (JPA/Hibernate)
[ Banco de Dados Relacional ]

[ Stack de Monitoramento: Actuator → Prometheus → Loki/Promptail → Grafana ]

O backend expõe endpoints REST para o frontend consumir. O banco de dados relacional armazena denúncias, usuários, logs de ações e demais entidades, com migrações gerenciadas por Flyway. Um conjunto de serviços auxiliares (Prometheus, Grafana, Loki, Promptail) compõe a camada de monitoramento.

Fluxo de dados

O fluxo de dados ocorre majoritariamente em três eixos: registro de denúncia, autenticação de usuários internos e envio de notificações.

[1] Aluno preenche o formulário de denúncia no frontend
  → [2] Frontend envia requisição POST /denuncias ao backend
    → [3] Backend valida dados, persiste no BD e gera código de acompanhamento
      → [4] Backend dispara evento interno para módulo de notificações (quando aplicável)
        → [5] Módulo de notificações encaminha e-mail/aviso para coordenador e, se permitido, para o aluno
  → [6] Frontend recebe resposta com código de acompanhamento e exibe ao aluno.

A autenticação de servidores coordenadores ocorre via módulo dedicado, comunicando-se com o provedor de identidade configurado (por exemplo, banco interno de usuários ou serviço externo).

Módulos funcionais

Módulo de denúncias

Responsável por todo o ciclo de vida da denúncia: criação, atualização de status, inclusão de comentários internos, anexos/mídias (se suportado) e auditoria.

Fluxo de criação e acompanhamento

[Aluno] → Frontend (formulário de denúncia)
  → POST /api/denuncias
    → Validações de negócio
    → Persistência com status "Recebida"
    → Geração de código de acompanhamento
    → Disparo de eventos para notificações
  → Resposta com código e status inicial

[Servidor] → Painel administrativo
  → GET /api/denuncias (filtro por status, data, etc.)
  → PUT /api/denuncias/{id} (atualização de status, comentários, etc.)
  → Histórico de ações armazenado em tabela própria (ex.: denuncia_log).

Módulo de Autenticação

Garante o login seguro de servidores coordenadores e outros perfis internos, utilizando, por exemplo, JWT ou sessão com cookies HTTP-only.

Fluxo de autenticação

[Servidor] → Frontend (tela de login)
  → POST /api/auth/login
    → Verificação de credenciais (e-mail + senha ou provedor externo)
    → Geração de token (JWT) com perfis/roles
    → Resposta para o frontend com token e dados básicos
  → Frontend armazena token com segurança (ex.: localStorage ou cookie seguro)
  → Todas as chamadas subsequentes incluem Authorization: Bearer <token>.

Módulo de Notificações

Centraliza o envio de notificações internas (para o painel) e externas (e-mail, eventualmente outros canais).

Fluxo de notificações internas e externas

[Backend] Evento de "Denúncia criada/atualizada"
  → Publicação em canal interno (ex: ApplicationEvent, fila, etc.)
    → Listener do módulo de notificações consome o evento
      → Gera notificações:
        • Internas: registro em tabela de notificações e exibição no painel
        • Externas: envio de e-mail/sms dependendo da configuração.

Como funciona o processo de migração com Flyway

O Flyway é usado para versionar o esquema do banco. Cada alteração é representada por um arquivo de migração incremental.

src/main/resources/db/migration/
├── V1__criacao_tabelas_iniciais.sql
├── V2__ajuste_campo_status_denuncia.sql
└── V3__tabela_notificacoes.sql

Ao iniciar a aplicação, o Flyway compara o histórico de migrações aplicadas no banco com os arquivos disponíveis e executa as pendentes.

Monitoramento

Como funciona o fluxo de monitoramento

O monitoramento da aplicação é realizado combinando métricas do Spring Boot Actuator, coleta pelo Prometheus, visualização via Grafana e logs centralizados em Loki, alimentados por Promptail (ou Promtail).

Spring Boot Actuator

O Actuator expõe endpoints de saúde, métricas e informações adicionais sobre a aplicação. Alguns endpoints comuns:

  • /actuator/health
  • /actuator/metrics
  • /actuator/prometheus

Prometheus

Responsável por scrapar as métricas expostas pelo Actuator e armazená-las para consulta.

No arquivo de configuração do Prometheus, existe um job configurado para o IFala apontando para o endpoint /actuator/prometheus do backend.

Promptail

Serviço responsável por coletar logs dos containers e encaminhar para o Loki. Verificar nome correto no projeto real (Promptail vs. Promtail).

Grafana & Loki

O Grafana é usado para visualização de métricas (Prometheus) e logs (Loki), através de dashboards configuráveis.

Deploy em produção

O deploy em produção pode ser automatizado via pipeline (CI/CD) e/ou apoiado em scripts, como o tui.sh.

Conteinerização com Docker

O projeto utiliza Docker para padronizar ambiente. Cada serviço (backend, frontend, banco, monitoramento) possui um Dockerfile ou é referenciado em docker-compose.yml.

# Build de imagem do backend (exemplo)
docker build -t ifala-backend:latest ./apps/ifala-backend

# Subir stack completa
docker compose up -d --build

Sobre o NGINX Gateway

O NGINX atua como gateway/reverso proxy, recebendo as requisições externas (HTTPS), roteando para o frontend (arquivos estáticos) e backend (API), além de aplicar regras de segurança básicas.

Exemplos de responsabilidades:

  • Redirecionamento HTTP → HTTPS.
  • Configuração de CORS, se necessário.
  • Rate limiting (se configurado).

Como realizar o deploy via arquivo tui.sh

O script tui.sh (Terminal User Interface) facilita operações comuns de deploy através de uma interface textual no terminal.

  1. Certificar-se de que o script possui permissão de execução.
chmod +x tui.sh
  1. Executar o script e seguir as opções do menu.
./tui.sh

O TUI pode incluir operações como: atualizar imagens, reiniciar containers, visualizar logs, verificar status de serviços, etc.

Sobre o Terminal User Interface (TUI)

O TUI é uma forma amigável de consolidar comandos repetitivos de operação em um menu textual, reduzindo erros e padronizando o processo de deploy e manutenção.
